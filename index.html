High 8 react-icon sub-libraries in use fa, fa6, fi, md, gi, pi, lu, tb, bi — 9 packages → consolidate to lucide-react

React Icons is a meta-package — each sub-library (fa6, md, fi…) resolves to a separate icon set with different visual styles, stroke weights, and design languages. The result is visually inconsistent UI across S3D. Standardizing on a single sub-library would allow tree-shaking and visual coherence.

react-icons/fa6 — FaChevronRight (Font Awesome, solid)
react-icons/fi — FiAlertTriangle (Feather, outlined, thinner)
react-icons/gi — GiResize (Game Icons — very different style)
react-icons/md — MdClose (Material Design, different grid)
react-icons/pi — PiMagicWand (Phosphor, duotone-capable)
react-icons/lu — LuTreeDeciduous (Lucide, clean modern)
High Legacy Font Awesome icons (CSS classes + custom wrapper) 121 hits across multiple files

Font Awesome CSS class icons (e.g. <i className="fa fa-close">) require a separate stylesheet to be loaded, are not tree-shakeable, and are visually inconsistent with component-based icons. All should be migrated to React components.

NavItems.tsx — fa-cube, fa-bar-chart, fa-share-alt, fa-arrow-left
LayersTreeItemCancel.tsx — fa-close
LayersTreeItemCancel (Layers2) — fa-close
DropdownModules.tsx — fa-fw (spacing utility only)
SidebarExports — fa-pulse (loading spinner via CSS)
High Duplicate SVG paths for same logical icon ~12 duplicated concepts

The same icon concept is imported from multiple different file paths across the codebase. This makes updates fragile — changing an icon requires hunting down all import paths. A centralized icon component registry would fix this.

Drone: ../Icons/drone.svg + @components/core/Icons/drone.svg + @styles/images/icons/drone-small.svg + relative paths
Edit: @components/core/Icons/edit.svg + @components/core/Icons/edit1.svg + @styles/images/icons/edit-pencil.svg
Fill: ./Fill.svg + @components/core/Icons/max-fill.svg + @styles/images/icons/max-fill.svg
Satellite: @components/core/Icons/satellite.svg + @styles/images/icons/satellite-small.svg + relative paths
Visibility: annotations-on.svg aliased as VisibleIcon (semantic mismatch)
Medium Semantic aliasing inconsistency ~8 mismatches

Some SVG files are imported under aliases that don't match their semantic meaning — making code harder to reason about and the icons harder to find during refactors.

VisibleIcon → annotations-on.svg (not a visibility icon)
ViewshedIcon → shade-on.svg (viewshed ≠ shade)
DuplicateSVG → copy.svg (fine, but CopySVG also exists)
RenameSVG → edit.svg (rename and edit are different actions)
KeepoutIcon → keepouts-on.svg (singular vs plural inconsistency)
Medium 7 hardcoded inline SVGs Arrow, PlusIcon, LightBulb, NavItems, PaneKeepOuts ×2, Viewshed

Inline SVGs embedded directly in JSX cannot be reused, themed, or audited easily. They also bloat component files. Each should be extracted to a shared icon component.

components/core/Icons/Arrow.tsx — full SVG path inline
components/core/Icons/PlusIcon.tsx — full SVG inline
components/core/Icons/IconLightBulb.tsx — inline with viewBox
components/Nav/NavItems.tsx:108 — inline SVG in nav config object
components/PaneKeepOuts/Options/AutoAI.tsx, FireSetback.tsx
Low SVG assets in 3 different directory trees @components/core/Icons, @styles/images/icons, @components/Toolbar/assets/icons

Custom SVG assets are scattered across three distinct directory trees, with no clear convention for which belongs where. This makes it hard to find icons and leads to duplication. Recommend a single @assets/icons/ source of truth.

@components/core/Icons/ — 30+ SVG files (most canonical)
@styles/images/icons/ — 15+ SVG files (style-adjacent)
@components/Toolbar/assets/icons/ — toolbar-specific
@components/ToolbarSimpleDesign/assets/icons/ — simple design toolbar
Plus local ./icons/ folders in ViewportOverlay, Compass
Low Stale react-icons/fa (v4) alongside react-icons/fa6 (v6) 3 fa + 23 fa6 usages

Font Awesome v4 icons (react-icons/fa) and v6 icons (react-icons/fa6) coexist. FaChevronRight and FaChevronDown appear in both. The fa package should be fully removed in favor of fa6, or ideally replaced with the target unified library.

Collapsible.tsx — FaChevronRight, FaChevronDown from react-icons/fa (v4)
ExpandCollapseButton — FaAngleUp, FaAngleDown from react-icons/fa6 (v6)
WidgetActionMenuContent — FaSave from react-icons/fa (v4)
